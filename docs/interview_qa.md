# AI 智能学习问答助手：前端面试问答准备

> 使用场景：面试官看不到具体代码时，用项目技术方案、实现思路、取舍和边界来回答。回答要真实，不要把项目包装成工业级复杂系统。

## 1. 项目整体介绍

### Q1：简单介绍一下这个项目。

**回答：**

这是一个基于大语言模型的 AI 学习问答平台，主要功能包括用户登录注册、多轮对话、历史会话管理、AI 回复流式输出、Markdown 渲染、代码高亮、头像上传以及聊天记录持久化。

前端使用 Vue3、TypeScript、Vite、Pinia、Vue Router 和 Element Plus；后端使用 Node.js、Express、TypeScript、MySQL，并通过后端统一代理 OpenAI 或 DeepSeek 等 AI 服务。整体是前后端分离架构，前端负责交互和流式渲染，后端负责鉴权、会话管理、AI 接口调用和数据持久化。

### Q2：这个项目你主要负责了哪些部分？

**回答：**

我主要负责前端整体架构和核心交互链路，包括 Vue3 项目搭建、页面组件拆分、Pinia 状态管理、Axios 请求封装、登录态维护、聊天流式输出、Markdown 渲染和会话管理。同时也参与了后端接口设计，包括 Express 路由、JWT 鉴权、SSE 流式响应、MySQL 数据表设计和 AI 接口代理。

可以重点强调：

- 前端：聊天页面、会话侧边栏、消息列表、输入框、登录注册、Markdown 渲染。
- 状态管理：用户登录态、当前会话、消息列表、生成状态、请求中断。
- 接口联调：普通 REST 接口 + SSE 流式接口。
- 后端协作：AI API 统一代理、MySQL 持久化、JWT 鉴权。

## 2. 技术栈相关

### Q3：为什么选择 Vue3 + TypeScript？

**回答：**

Vue3 的 Composition API 更适合按业务逻辑组织代码，比如聊天输入、会话切换、消息渲染、用户状态都可以拆成更清晰的状态和方法。TypeScript 可以增强类型约束，尤其是这个项目里有用户、会话、消息、AI 响应事件等多种数据结构，用 TS 可以减少接口字段不一致、状态类型错误等问题。

另外 Vue3 的响应式系统和 Pinia 配合比较自然，适合处理聊天消息实时追加、生成状态变化这类动态 UI。

### Q4：为什么用 Pinia，而不是只用组件状态？

**回答：**

因为聊天系统里有很多跨组件共享状态，例如当前用户、token、当前会话、消息列表、是否正在生成、会话列表等。如果全部放在组件内部，会出现多层 props 传递和兄弟组件通信复杂的问题。

我用 Pinia 拆了两类核心状态：

- 用户状态：token、用户信息、登录状态。
- 聊天状态：会话列表、当前会话、消息列表、生成状态、中断控制器。

这样页面组件只负责展示和触发操作，核心业务状态集中在 store 里，结构更清晰，也方便后续维护。

### Q5：为什么普通请求用 Axios，流式聊天接口不用 Axios？

**回答：**

普通登录、注册、会话列表、重命名、删除这些接口都是标准 REST 请求，所以用 Axios 封装拦截器比较方便，可以统一加 token、统一处理 401。

但是聊天回复需要流式输出，需要直接读取浏览器的 `ReadableStream`。所以流式聊天接口使用原生 `fetch`，通过 `response.body.getReader()` 读取数据流，再用 `TextDecoder` 解码。这样前端可以边接收 token 边更新页面，实现逐字输出效果。

## 3. 流式输出与 SSE

### Q6：AI 回复的流式输出是怎么实现的？

**回答：**

整体链路是：

1. 用户在前端输入问题并发送。
2. 前端先在本地创建用户消息和一条空的 AI 回复消息。
3. 前端通过 fetch 请求后端的流式接口。
4. 后端调用 AI 服务的 stream 模式，逐步接收模型返回的 token。
5. 后端通过 SSE 把 token 持续推给前端。
6. 前端收到 token 后追加到当前 AI 消息内容中。
7. 流结束后，前端把消息状态改为完成，并刷新会话数据。

这里的关键点是后端返回 `text/event-stream`，每次写出一段 `data` 事件；前端持续读取响应流，并解析事件内容。

### Q7：为什么选择 SSE，而不是 WebSocket？

**回答：**

这个项目的聊天流式输出主要是服务端持续向客户端推送 token，本质是单向推送。SSE 天然适合这种场景，实现成本比 WebSocket 低，也更贴合 HTTP 请求响应模型。

WebSocket 更适合双向实时通信，比如多人协作、在线状态、实时游戏、客服系统等。当前项目不需要客户端和服务端长期双向通信，所以 SSE 更简单、稳定，也方便和现有 REST 接口共存。

### Q8：SSE 数据被拆包怎么处理？

**回答：**

前端不能假设每次读取到的数据都是完整事件，因为网络传输可能把一个事件拆成多段，也可能一次返回多个事件。

所以我会维护一个缓冲区，把每次读取到的 chunk 追加进去，然后按 SSE 的事件分隔符进行切分。完整事件就解析处理，不完整的部分保留在缓冲区，等待下一次数据到来再继续拼接。

### Q9：用户点击停止生成时怎么处理？

**回答：**

前端使用 `AbortController` 控制当前 fetch 请求。用户点击停止时，前端调用 abort，中断当前请求，并把本地正在生成的 AI 消息状态更新为完成或停止。

后端也会监听连接关闭状态，在流式生成过程中检查连接是否已经断开。如果客户端已经中断，就停止继续向前端写数据。这样可以避免前端已经不需要结果时，后端还继续推送。

如果做得更完善，还可以增加模型侧请求取消、停止状态落库、超时控制和生成任务队列。

### Q10：如何保证流式输出时页面不卡？

**回答：**

主要从几个方面处理：

- token 到达后只更新当前正在生成的那条 AI 消息，而不是重建整个消息列表。
- 消息状态集中在 Pinia 中，组件基于响应式数据自动更新。
- Markdown 渲染只针对消息内容进行展示，避免无关组件重复渲染。
- 生成过程中限制重复发送，避免多个流式请求同时修改同一份状态。

如果后续消息量很大，可以进一步做虚拟列表、节流渲染和按会话分页加载。

## 4. 状态管理与交互

### Q11：发送一条消息后，前端状态是怎么变化的？

**回答：**

发送消息时，前端会先校验内容是否为空，以及当前是否正在生成。如果没有当前会话，就先创建新会话。

然后前端会做乐观更新：

1. 添加一条用户消息。
2. 添加一条空的 AI 回复消息，状态设为生成中。
3. 设置全局生成状态为 true。
4. 发起流式请求。
5. 每收到一个 token，就追加到 AI 消息内容中。
6. 收到完成事件后，把消息状态改成完成。
7. 最后清除生成状态和中断控制器。

这样用户不需要等后端完整返回，页面能立即给出反馈。

### Q12：为什么要做乐观更新？

**回答：**

如果等后端完整返回再更新 UI，用户会感觉发送后没有反馈。乐观更新可以让用户消息立刻出现在页面上，同时马上显示一条正在生成的 AI 消息，提高交互体验。

当然乐观更新要处理失败场景。如果请求失败，需要把 AI 消息状态标记为失败，并给用户错误提示。

### Q13：如何防止用户重复发送？

**回答：**

项目里通过生成状态控制。当 `isGenerating` 为 true 时，前端会禁止再次发送，避免多个流式请求同时进行，导致消息顺序错乱或多个 token 同时写入。

也可以在输入框按钮上做 loading 和 disabled 状态，让用户明确知道当前正在生成。

### Q14：切换会话时，如果当前正在生成怎么办？

**回答：**

切换会话前会先停止当前生成请求，避免旧会话的流式响应继续写入当前页面。然后再请求目标会话详情，更新当前会话 ID 和消息列表。

这个处理可以避免一个常见问题：用户切到新会话后，旧请求返回的 token 还在追加，造成消息串会话。

## 5. 登录鉴权

### Q15：登录鉴权是怎么实现的？

**回答：**

后端登录成功后签发 JWT，前端保存 token 和用户信息。之后普通请求通过 Axios 请求拦截器自动携带 `Authorization: Bearer token`，流式请求因为用 fetch，也会手动带上 token。

后端通过鉴权中间件解析 token，校验成功后把用户信息挂到请求对象上，后续接口就可以根据用户 ID 查询对应会话和消息。

### Q16：token 存在 localStorage 安全吗？

**回答：**

localStorage 的优点是实现简单，刷新页面后也能恢复登录态。但它有 XSS 风险，如果页面被注入恶意脚本，token 可能被读取。

这个项目是学习项目，所以采用了 localStorage。生产环境更推荐使用 HttpOnly Cookie 存储 token，配合 SameSite 和 CSRF 防护；或者使用 access token + refresh token 的方案。

### Q17：401 过期怎么处理？

**回答：**

前端在响应拦截器中统一处理 401。如果后端返回未登录或 token 过期，前端会清除本地 token 和用户信息，并跳转到登录页。

这样可以避免每个业务接口都单独写一遍登录过期处理逻辑。

## 6. Markdown 与代码高亮

### Q18：AI 回复里的 Markdown 是怎么渲染的？

**回答：**

前端使用 Markdown 渲染库把 AI 返回的 Markdown 文本转换成 HTML，再通过组件展示。代码块部分接入 highlight.js，根据语言标识做代码高亮。

这样 AI 回复中的标题、列表、代码块、换行等格式都能比较友好地展示，适合学习问答场景。

### Q19：使用 v-html 会不会有 XSS 风险？

**回答：**

会有风险，所以项目里需要限制 Markdown 渲染配置。我的处理思路是关闭原始 HTML 渲染，只允许 Markdown 语法转换；对代码块内容进行 HTML 转义，避免脚本被执行。

如果是生产项目，我会进一步接入 DOMPurify，对最终 HTML 做二次净化，增强安全性。

### Q20：为什么要做代码高亮？

**回答：**

这个项目定位是学习问答助手，AI 经常会返回代码示例。如果没有代码高亮，大段代码可读性比较差。接入代码高亮后，用户能更容易区分关键字、字符串、注释和结构，学习体验更好。

## 7. 后端与 AI 服务代理

### Q21：为什么前端不直接调用 OpenAI 或 DeepSeek？

**回答：**

主要有三个原因：

1. API Key 不能暴露在浏览器里。
2. 后端可以统一做鉴权、限流、日志和错误处理。
3. 后端可以屏蔽不同模型供应商的差异，前端只需要调用统一接口。

所以项目里前端只调用自己的后端接口，由后端根据配置选择 OpenAI 或 DeepSeek。

### Q22：OpenAI 和 DeepSeek 是怎么切换的？

**回答：**

后端通过环境变量配置当前 AI provider。如果配置为 OpenAI，就走 OpenAI 服务；如果配置为 DeepSeek，就走 DeepSeek 服务。

因为 DeepSeek 接口兼容 OpenAI SDK 的调用格式，所以后端可以复用一部分封装逻辑，只需要替换 baseURL、API Key 和模型名称。

### Q23：AI 接口调用失败怎么办？

**回答：**

后端会捕获 AI 调用异常，并通过 SSE 返回错误事件。前端收到错误事件后，把当前 AI 消息状态标记为失败，并展示错误提示。

同时后端会把已经生成的部分内容保存下来。如果完全没有内容，则可以标记为失败；如果已经生成了一部分，可以保留这部分内容，并记录错误信息。

## 8. MySQL 数据设计

### Q24：数据库表是怎么设计的？

**回答：**

核心是三类数据：

- 用户表：保存用户名、邮箱、密码哈希、头像等信息。
- 会话表：保存会话标题、所属用户、模型、更新时间、删除状态。
- 消息表：保存具体消息，包括所属会话、角色、内容、模型、生成状态、错误信息等。

关系上是：

- 一个用户可以有多个会话。
- 一个会话可以有多条消息。

### Q25：为什么会话删除用软删除？

**回答：**

会话是用户的重要历史数据，软删除可以避免误删后无法恢复，也方便后续做回收站、数据审计或恢复功能。

实现上是给会话加 `deleted_at` 字段，删除时不真正删除记录，而是更新删除时间；查询列表时过滤掉已删除的数据。

### Q26：怎么防止 SQL 注入？

**回答：**

后端数据访问层使用参数化 SQL，也就是用占位符传参，而不是拼接用户输入。这样数据库驱动会处理参数转义，避免用户输入被当作 SQL 语句执行。

另外接口层会对参数做基础校验，比如会话 ID、消息内容、登录字段等。

### Q27：聊天记录如何加载？

**回答：**

进入聊天页后，前端先加载当前用户的会话列表。用户选择某个会话时，再请求该会话详情和消息列表。消息按创建时间排序，保证历史对话按发送顺序展示。

如果数据量进一步增大，可以把消息列表改成分页或游标加载，避免一次性加载过多历史消息。

## 9. 会话标题自动生成

### Q28：会话标题自动生成是怎么做的？

**回答：**

当用户在一个新会话里发送第一条消息时，后端会根据这条消息生成一个简短标题。优先调用 AI 生成标题，如果失败，就使用本地兜底规则，比如截取用户问题的前几个字作为标题。

这样用户不用手动命名，会话列表也比默认的“新会话”更容易识别。

### Q29：为什么标题生成放在后端？

**回答：**

因为标题生成需要调用模型接口，放在后端可以避免暴露 API Key，也方便统一控制模型、提示词和失败兜底逻辑。前端只需要刷新会话列表即可。

## 10. 错误处理与边界

### Q30：项目里做了哪些错误处理？

**回答：**

主要包括：

- Axios 响应拦截器统一处理 401。
- 后端接口参数校验，避免非法参数进入业务逻辑。
- 后端统一错误处理中间件处理异常响应。
- 流式接口通过 error 事件把 AI 调用错误返回给前端。
- 前端对失败消息做状态标记，并给用户提示。

### Q31：如果网络中断，用户体验如何？

**回答：**

如果流式请求中断，前端会停止继续接收 token，并把当前消息状态更新。后端也会检测连接关闭，不再继续写响应。

当前项目更偏基础实现。如果要优化，可以增加“重新生成”“继续生成”“失败重试”等能力，让用户可以从中断处恢复。

### Q32：这个项目目前有什么不足？

**回答：**

我认为主要有几个不足：

1. 缺少完整的自动化测试，比如组件测试、接口测试和端到端测试。
2. 流式输出虽然实现了，但还可以增加更细的状态，比如 stopped、timeout、retry。
3. 消息列表目前适合中小量数据，历史消息很多时需要分页或虚拟列表。
4. token 存储在 localStorage，生产环境可以改成 HttpOnly Cookie。
5. Markdown 安全可以进一步加 DOMPurify。
6. AI 调用可以增加限流、审计日志和成本统计。

这个回答很重要，能体现你不是只会吹项目，也知道如何改进。

## 11. 性能优化

### Q33：这个项目做过哪些性能优化？

**回答：**

主要是交互层面的优化：

- 使用 SSE 实现流式输出，降低用户等待完整响应的体感时间。
- 前端只更新当前生成中的消息，减少不必要的状态改动。
- 会话和消息分开加载，不在首页一次性加载所有历史内容。
- 使用 Vite 提升开发构建速度。
- 通过 Pinia 集中状态，减少组件之间复杂通信。

如果要继续优化，可以做消息虚拟列表、Markdown 渲染缓存、长会话分页加载、接口缓存和资源按需加载。

### Q34：你简历里说提升用户体验，具体体现在哪里？

**回答：**

主要体现在三个方面：

1. 流式输出：用户不用等完整回答生成完，能马上看到 AI 开始回复。
2. 本地乐观更新：发送后用户消息立即显示，页面反馈更及时。
3. Markdown 和代码高亮：AI 返回内容更适合阅读，尤其是代码类回答。

另外还有历史会话管理、自动标题、停止生成、重新生成等交互能力。

## 12. 前端工程化

### Q35：项目结构是怎么划分的？

**回答：**

前端按职责分层：

- pages：页面级组件，比如登录页、聊天页。
- components：可复用组件，比如消息列表、输入框、会话侧边栏、Markdown 渲染器。
- stores：Pinia 状态管理。
- services：接口请求封装。
- router：路由和登录守卫。
- types：前端数据类型定义。

这样结构比较清晰，组件、状态和接口职责分离。

### Q36：Axios 封装做了什么？

**回答：**

Axios 主要用于普通 REST 接口。封装里做了请求拦截和响应拦截：

- 请求拦截：自动从本地读取 token，并添加到请求头。
- 响应拦截：如果遇到 401，统一清除登录态并跳转登录页。

这样业务代码不用每次手动处理 token 和登录过期。

### Q37：路由守卫怎么做？

**回答：**

路由配置里给需要登录的页面加 `requiresAuth` 标识。进入路由前检查 Pinia 里的登录状态，如果没有登录就跳到登录页；如果已登录访问登录页，就跳回聊天页。

这可以避免未登录用户直接访问聊天页面。

## 13. 高频追问

### Q38：如果让你重新设计这个项目，你会怎么优化？

**回答：**

我会从几个方面优化：

- 安全：token 改成 HttpOnly Cookie，Markdown 加 DOMPurify。
- 性能：消息列表加虚拟滚动，历史消息分页加载。
- 稳定性：SSE 增加超时、重试、停止状态和错误恢复。
- 工程化：补充单元测试、接口测试和 E2E 测试。
- 业务能力：增加模型切换、提示词模板、收藏会话、搜索历史记录。
- 可观测性：记录首 token 时间、总响应时间、错误率和 token 消耗。

### Q39：你在项目中遇到过什么难点？

**回答：**

比较大的难点是流式输出的状态管理。因为它不是一次请求一次响应，而是持续返回 token。前端要处理生成中、完成、失败、停止等状态，还要保证 token 追加到正确的消息上。

我的解决方式是把聊天状态集中放到 Pinia 中，发送消息时创建本地 assistant 占位消息，后续 token 都追加到这条消息。这样组件只负责渲染，复杂逻辑集中在 store 和接口层。

### Q40：你怎么证明这个项目是你做的？

**回答：**

我可以从完整业务链路讲清楚，比如用户发送消息后，前端如何创建本地消息、如何发起 fetch 流式请求、后端如何设置 SSE、如何调用 AI stream、如何把 token 推回前端、如何落库、如何更新会话标题。

这些不是只看页面能背出来的，需要了解前后端状态流和数据流。

## 16. 三分钟项目介绍模板

**回答模板：**

这个项目是一个面向学习场景的 AI 问答平台，用户登录后可以创建多个会话，进行多轮 AI 对话。AI 回复支持流式输出，内容支持 Markdown 渲染和代码高亮，历史会话和聊天消息会保存到 MySQL。

前端部分我使用 Vue3 + TypeScript + Vite 搭建，UI 使用 Element Plus。状态管理用 Pinia，拆成用户状态和聊天状态。用户状态负责 token、用户信息和登录状态；聊天状态负责会话列表、当前会话、消息列表、是否正在生成以及停止生成控制。普通接口使用 Axios 封装请求拦截器和响应拦截器，自动携带 token，并统一处理 401。聊天流式接口使用 fetch，因为需要读取 ReadableStream。

流式输出是项目重点。用户发送消息后，前端会先本地添加用户消息和一条空的 AI 回复消息，形成即时反馈。然后请求后端 SSE 接口。后端设置 `text/event-stream` 响应头，调用 AI 服务的 stream 模式，拿到 token 后通过 SSE 写回前端。前端不断解析事件，把 token 追加到当前 AI 消息中。生成完成后，再刷新会话数据。

后端使用 Express + TypeScript，核心模块包括认证、会话、消息和 AI 服务代理。认证使用 JWT，密码使用 bcrypt 哈希。数据层使用 MySQL，主要有用户表、会话表和消息表。AI 服务通过后端统一代理，可以通过环境变量切换 OpenAI 或 DeepSeek，前端不会直接接触 API Key。

这个项目目前更偏学习和实战展示。它的亮点是完整打通了前端交互、SSE 流式响应、AI API 调用和 MySQL 持久化。后续如果继续优化，我会补充消息分页、虚拟列表、DOMPurify 安全过滤、HttpOnly Cookie 登录方案、接口测试和 E2E 测试。
